# 协作说明

这份文档用于后续新对话快速对齐项目习惯。

## 先看什么

1. `README.md`
2. 本文件
3. `docs/项目状态与目标.md`
4. 当前任务对应的 `docs/superpowers/` 计划或设计文档

## 提交规范

- 每次功能变更完成后，需要同步提交对应 commit，不要只停留在聊天或本地未提交改动里。
- commit message 使用规范类型前缀，类型保持英文，例如 `feat:`、`fix:`、`docs:`、`test:`、`refactor:`、`chore:`。
- commit message 冒号后的摘要和详细说明使用中文，例如 `feat: 完善创作者文章预览体验`。
- 提交前后都需要说明本次提交包含的核心改动和验证结果。
- 提交前先确认 `git status`，不要把无关文件一起提交。

## 命名规范

- 文件名尽量使用中文，尤其是文档、说明和计划类文件。
- 代码文件如果已经有稳定引用，就不要为了中文名强行改动。

## 工作区规范

- 不要擅自回滚别人的未确认改动。
- 数据库改动优先写增量迁移脚本。
- 数据库增量脚本由用户手动执行；协作过程中只生成、校验和提交脚本，不代为执行，也不直接修改数据库。
- 如果后续验证依赖尚未执行的数据库变更，先暂停并提示用户执行，待用户确认后再继续。
- 需要长期保留的约定，优先写进 `README.md` 或本文件。
- 功能方向变动、功能完成或新增/删减时，先同步更新 `docs/项目状态与目标.md`，再去改对应计划或代码。

## 启动约定

- 后端启动命令：`cd blog-backend && mvn.cmd spring-boot:run`
- 前端启动命令：`cd blog-frontend && npm.cmd run dev -- --host 127.0.0.1`
