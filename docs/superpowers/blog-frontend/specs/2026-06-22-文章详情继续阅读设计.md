# 文章详情继续阅读设计

## 背景

第一阶段已经完成博客首页和文章详情页的基础阅读体验增强。首页现在承担文章发现入口，详情页已经具备标题区、封面、阅读信息、目录和正文阅读结构。

当前详情页还缺少两个关键能力：

1. 读者读完一篇文章后，没有自然的下一步阅读路径。
2. 长文章阅读时，回到顶部按钮必须滚动到底部才能看到，目录也没有随着阅读位置实时反馈。

因此第二阶段先做“继续阅读”和“长文阅读辅助”这一组能力，不做评论、点赞、收藏等互动功能。

## 设计目标

1. 让文章详情页读完后有明确去处：可以继续读上一篇、下一篇或相关文章。
2. 保证所有公开推荐内容都只来自已发布、审核通过、未删除的文章。
3. 让长文阅读时的目录和回顶操作更顺手，减少滚动成本。
4. 不新增数据库表，不引入评论、点赞、收藏等需要额外业务规则的功能。
5. 保持现有清爽、偏阅读型的视觉方向，不把详情页做成信息过载的门户页。

## 本次范围

### 上一篇和下一篇

文章详情页底部展示上一篇和下一篇公开文章。

规则如下：

1. 上一篇表示发布时间早于当前文章的最近一篇公开文章。
2. 下一篇表示发布时间晚于当前文章的最近一篇公开文章。
3. 如果同一发布时间存在多篇文章，使用文章编号作为稳定排序补充。
4. 当前文章不存在、未发布、未审核通过或已删除时，不返回相邻文章。
5. 相邻文章只展示轻量信息：文章编号、标题、创建时间、阅读数、作者名称和标签。

### 相关文章

文章详情页底部展示与当前文章标签相同的公开文章。

规则如下：

1. 排除当前文章。
2. 只推荐已发布、审核通过、未删除的文章。
3. 优先按相同标签数量排序，相同数量时按发布时间倒序排序。
4. 默认最多展示四篇。
5. 如果当前文章没有标签，或没有匹配文章，则不展示相关文章卡片，只保留页面自然收尾。

### 悬浮回到顶部

当前正文底部的“回到顶部”按钮改为右侧悬浮按钮。

规则如下：

1. 页面向下滚动一段距离后显示。
2. 点击后平滑滚动回页面顶部。
3. 桌面端放在页面右下侧，不遮挡正文和目录。
4. 移动端放在右下角，并避开常见底部安全区域。
5. 按钮使用图标或短文案，不做大面积视觉干扰。

### 目录跟随和当前高亮

桌面端目录保持右侧吸附，滚动正文时当前章节高亮。

规则如下：

1. 目录继续来源于文章中的二级和三级标题。
2. 页面滚动时，根据当前可视区域内的标题更新高亮项。
3. 当前目录项使用主色或更清晰的背景态区分。
4. 点击目录项仍然平滑滚动到对应标题。
5. 移动端目录继续放在正文前，不做悬浮目录，避免遮挡阅读内容。

## 不做范围

本次不做以下功能：

1. 评论。
2. 点赞。
3. 收藏。
4. 阅读趋势统计。
5. 热门排行。
6. 新增数据库表。
7. 新增登录后才可用的读者互动能力。

这些能力后续可以单独设计，避免一次性引入过多权限、审核和数据一致性问题。

## 后端设计

### 相邻文章接口

新增公开接口，用于获取当前文章的上一篇和下一篇。

路径建议：

```text
/api/articles/{id}/neighbors
```

返回结构建议：

```json
{
  "previous": {
    "id": 1,
    "title": "更早发布的文章",
    "createdAt": "2026-06-01T10:00:00",
    "viewCount": 12,
    "authorName": "作者",
    "tags": []
  },
  "next": {
    "id": 3,
    "title": "更新发布的文章",
    "createdAt": "2026-06-03T10:00:00",
    "viewCount": 18,
    "authorName": "作者",
    "tags": []
  }
}
```

如果某一侧不存在文章，对应字段返回空值。

### 相关文章接口

新增公开接口，用于获取当前文章的相关文章。

路径建议：

```text
/api/articles/{id}/related?size=4
```

返回结构建议为文章摘要列表，字段与公开文章列表保持一致或更轻量。

查询规则：

1. 先确认当前文章本身是公开可见文章。
2. 查询当前文章的标签。
3. 根据相同标签数量对其他公开文章排序。
4. 排除当前文章。
5. 按传入数量限制返回结果，默认四篇。

### 服务边界

相邻文章和相关文章都属于公开阅读能力，应放在现有公开文章服务中，不进入后台管理接口。

后端应复用当前公开文章可见性规则，保证未发布、未审核通过、已删除的文章无论是否登录都不会被推荐出来。

## 前端设计

### 数据流

文章详情页加载当前文章成功后，同时加载：

1. 相邻文章。
2. 相关文章。

当前文章加载失败时，不请求推荐数据。

推荐数据加载失败时，不影响正文阅读，只隐藏对应推荐区或显示轻量空状态。

### 页面结构

详情页结构调整为：

1. 返回按钮。
2. 文章标题区。
3. 阅读摘要。
4. 正文和目录双栏阅读区。
5. 继续阅读区。
6. 右侧悬浮回顶按钮。

继续阅读区放在正文结束之后，作为读完文章后的自然收尾。

### 继续阅读区

继续阅读区由两部分组成：

1. 上一篇 / 下一篇导航。
2. 相关文章卡片列表。

上一篇和下一篇使用横向双卡布局：

1. 左侧为上一篇。
2. 右侧为下一篇。
3. 缺少某一侧文章时，保留另一侧，不显示空卡片。

相关文章使用最多四张紧凑卡片：

1. 标题是主要信息。
2. 辅助展示发布时间、阅读数和标签。
3. 点击卡片进入对应文章详情页。

### 目录和回顶

桌面端右侧目录继续吸附在顶部下方，并增加当前章节高亮。

悬浮回顶按钮独立于正文底部展示，不再要求读者滚动到底部才能使用。按钮只在用户向下阅读后出现，减少首屏干扰。

## 视觉方向

本次视觉关键词是“阅读路径”。

正文区域仍然保持安静，新增能力主要放在正文之后和右侧辅助区：

1. 右侧目录负责“现在读到哪里”。
2. 悬浮回顶负责“快速回到开头”。
3. 底部继续阅读负责“接下来读什么”。

继续阅读区可以比普通正文卡片稍微更有结构感，但不使用过重装饰。它应该像文章自然结尾处的一组阅读建议，而不是广告区。

## 异常和空状态

1. 当前文章不可见时，详情页保持现有错误提示。
2. 相邻文章接口失败时，不影响文章正文展示。
3. 相关文章接口失败时，不影响文章正文展示。
4. 没有上一篇或下一篇时，只展示存在的一侧。
5. 没有相关文章时，不展示相关文章模块。
6. 没有目录时，不展示目录模块，也不出现当前目录高亮逻辑。

## 验证方式

实现后需要验证：

1. 公开文章详情页能正常显示上一篇和下一篇。
2. 未发布、未审核通过、已删除的文章不会出现在上一篇、下一篇和相关文章中。
3. 相关文章按相同标签推荐，并排除当前文章。
4. 没有相邻文章或相关文章时，页面不出现空白异常区域。
5. 文章较长时，回顶按钮在滚动后悬浮显示，点击能回到顶部。
6. 桌面端目录吸附在右侧，滚动时当前章节高亮正确变化。
7. 移动端正文、目录、继续阅读区和回顶按钮不互相遮挡。
8. 前端生产构建通过。
9. 后端相关单元测试通过。
