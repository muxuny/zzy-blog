# 阅读位置与进度设计

## 背景

“我的阅读与阅读历史”第一阶段已经完成：登录用户打开公开文章后会记录首次阅读时间、最近阅读时间和阅读次数，并能在“我的阅读”和完整历史页继续最近一篇可用文章。

当前“继续阅读”只会重新打开文章详情页，不会保存或恢复读者在正文中的位置。对于长文章，读者从“我的阅读”返回时仍需要手动寻找上次读到的位置。本期在现有登录用户阅读历史基础上补齐阅读位置和阅读进度。

## 目标

- 为登录用户保存公开文章的最近阅读位置和阅读进度。
- 进入文章详情后展示轻量“继续上次阅读”提示，由用户点击恢复位置。
- 在“我的阅读”和“阅读历史”中展示阅读进度，例如“读到 42%”。
- 文章内容更新后不再使用旧位置恢复，避免跳转到错误位置。
- 沿用现有“我的阅读”信息架构，不新增游客入口或公开首页已读状态。

## 非目标

本期不实现以下能力：

- 游客本地阅读位置或游客阅读历史。
- 多个书签、章节书签或手动保存按钮。
- 自动跳转到上次位置。
- “已读完”状态、完成率统计或阅读成就。
- 公开首页文章卡片的“已读”标签。
- 阅读趋势、热门排行、评论或点赞。

## 方案选择

采用扩展 `article_reading_history` 的方案。阅读历史仍表示“用户读过某篇文章”，同时保存该关系的最近阅读位置。

备选方案包括：

- 新增独立 `article_reading_position` 表：边界更清晰，但本期聚合和接口复杂度更高。
- 前端本地存储：实现快，但不支持登录用户跨设备恢复，也无法稳定展示在“我的阅读”和历史页。

当前项目已经有登录用户维度的阅读历史表、概览接口和历史列表，扩展现有表能直接服务本期目标，并保持实现范围可控。

## 数据模型

在 `article_reading_history` 上通过增量脚本新增字段：

- `progress_percent INT NOT NULL DEFAULT 0`：阅读进度，范围 0 到 100。
- `scroll_y INT NOT NULL DEFAULT 0`：页面纵向滚动位置，用于恢复。
- `anchor_id VARCHAR(160) DEFAULT NULL`：当前最近标题锚点，作为辅助定位信息。
- `anchor_offset INT DEFAULT NULL`：当前位置相对锚点偏移，作为辅助定位信息。
- `article_updated_at_snapshot DATETIME DEFAULT NULL`：保存位置时的文章更新时间快照。
- `position_updated_at DATETIME DEFAULT NULL`：最近保存位置时间。

`position_updated_at` 和 `article_updated_at_snapshot` 是业务字段。`created_at` 和 `updated_at` 继续只承担审计含义，不能复用为阅读位置业务时间。

保存位置时更新上述字段和审计字段。删除历史后再次阅读时，沿用现有重新激活规则，并将位置字段重置为默认值，避免恢复删除前的位置。

## 后端设计

`ArticleController` 继续负责公开详情读取。登录用户成功读取公开文章时仍记录阅读历史，但不会在详情读取阶段保存位置。

`ReadingHistoryService` 扩展以下职责：

- 保存当前用户指定文章的阅读位置。
- 在历史条目中输出阅读进度和恢复状态。
- 判断当前文章是否仍可按旧位置恢复。

新增需登录的阅读位置保存接口：

```text
PUT /api/my/reading/history/{articleId}/position
```

请求体：

```json
{
  "progressPercent": 42,
  "scrollY": 3810,
  "anchorId": "section-2",
  "anchorOffset": 120,
  "articleUpdatedAt": "2026-07-21T16:40:00"
}
```

接口规则：

- 仅当前登录用户可保存自己的阅读位置。
- 只能更新当前用户已有且未删除的阅读历史，不因保存位置额外创建历史。
- 保存前校验文章仍是未删除、已发布且公开可见。
- `progressPercent` 限制为 0 到 100。
- `scrollY` 不小于 0。
- `anchorId` 可为空，非空时限制长度。
- `anchorOffset` 可为空，非空时不小于 0。
- 保存失败时接口返回失败；文章详情阅读本身不受影响。

`ReadingHistoryItem` 增加输出字段：

- `progressPercent`
- `positionUpdatedAt`
- `canResume`
- `resumeScrollY`
- `resumeAnchorId`
- `resumeAnchorOffset`

恢复判断规则：

- 只对可用文章计算恢复状态。
- `progressPercent >= 5` 时才允许提示恢复。
- `article_updated_at_snapshot` 与当前文章 `updated_at` 匹配时才允许恢复。
- 不可公开历史仍只输出标题快照、阅读时间、阅读次数、进度和不可用提示，不输出正文、摘要、封面、作者等当前文章信息，也不允许恢复。

## 前端设计

### 文章详情页

登录用户进入公开文章详情后，如果后端返回 `canResume=true`，在标题区或正文上方展示轻量提示：

```text
上次读到 42%，继续阅读
```

用户点击“继续阅读”后滚动到保存位置。页面不会自动跳转。提示提供关闭按钮，本次页面生命周期内关闭后不再显示。

游客不展示恢复提示，也不保存阅读位置。保存位置失败静默处理，不弹全局错误提示，避免打扰阅读。

### 位置计算与保存

前端按正文阅读区域计算 `progressPercent`，并限制在 0 到 100。保存内容包括：

- 当前进度百分比。
- 当前 `scrollY`。
- 当前目录高亮标题 ID。
- 相对标题偏移。
- 当前文章 `updatedAt` 快照。

滚动时节流自动保存，建议每 8 到 10 秒最多保存一次。页面隐藏、路由离开或组件卸载时补保存一次。

恢复位置时本期优先使用 `scrollY`。`anchorId` 和 `anchorOffset` 作为辅助数据保留，后续如果发现布局高度变化明显，可以再用于兜底定位。

### 我的阅读

`/reading` 保持现有三段结构：

- “上次阅读”显示“读到 42%”。
- “最近阅读”每条显示“读到 42%”。
- 不可公开记录可以显示进度，但仍置灰且不可点击。

### 阅读历史

`/reading/history` 在阅读时间和阅读次数旁展示“读到 42%”。单条删除、清空、分页和时间分组逻辑保持不变。

不新增“已读完”标签，也不在公开首页文章卡片展示“已读”。

## 异常边界

- 用户未登录或 token 失效：不保存位置；公开文章正文仍按现有规则可读。
- 文章变为私有、草稿或删除：保存位置失败，历史和概览继续按不可公开快照规则展示。
- 文章内容更新：仍可展示历史进度，但不提示恢复，也不使用旧位置跳转。
- 低于 5%：可以保存和展示进度，但不提示恢复。
- 接近底部：最高显示到 100%，不自动标记已读完。
- 高频滚动：前端节流；后端只接受合法值并限定当前用户。
- 多标签页：后写入的位置覆盖先写入的位置，暂不做冲突提示。
- 删除历史后再次阅读：历史重新激活，位置字段重置为默认值。

## 测试与验收

后端测试覆盖：

- 数据库增量脚本和 `init.sql` 中新增字段一致。
- 新增字段不复用 `created_at` 或 `updated_at` 作为业务时间。
- 保存位置参数校验。
- 只能更新当前用户已有历史，不能跨用户更新。
- 不允许为不可公开文章保存位置。
- 内容更新时间快照匹配时 `canResume=true`，不匹配时 `canResume=false`。
- 低于 5% 不可恢复但可展示进度。
- 删除历史后重新阅读会重置位置字段。

前端测试覆盖：

- 阅读位置 API 请求路径、方法和参数。
- 登录且 `canResume=true` 时文章详情展示恢复提示。
- 点击提示后滚动到保存位置。
- 关闭提示后本次页面生命周期内不再显示。
- 游客或 `canResume=false` 不展示恢复提示。
- 滚动保存节流与离开前补保存。
- “我的阅读”和“阅读历史”展示“读到 xx%”。
- 不可用记录不泄露文章详情字段，不允许恢复。
